# [스킬] 봇·프로세스 운영 표준 (Bot Operations)

## 1. 발동 조건

다음 중 하나에 해당하면 이 스킬을 우선 적용한다:

- 장기 실행 프로세스(트레이딩 봇, Hermes Agent, dashboard, cloudflared, watchdog 등)를 **시작·재시작·교체·검증**할 때
- 외부 주체(봇, 디스패치, 에이전트, LLM)의 **발언이 사실인지 확인**이 필요할 때
- **다중 봇 상태**를 동시에 보고해야 할 때
- 사용자가 "X 실행해줘", "Y 재시작해줘", "Z 상태 알려줘"라고 요청할 때

## 2. 준수 설계 규칙 (레시피)

### 2.1 프로세스 수명주기 (Lifecycle)

모든 프로세스 교체는 다음 순서를 **반드시** 강제한다:

1. **기존 프로세스 식별**
   ```bash
   ps aux | grep <name> | grep -v grep
   ```
2. **정상 종료 시도** (SIGTERM)
   ```bash
   kill <PID>
   ```
3. **10초 대기 후 잔존 확인 → 강제 종료** (SIGKILL)
   ```bash
   sleep 10 && ps -p <PID> && kill -9 <PID>
   ```
4. **새 소스 배포 / config 갱신** (백업 동반 — 2.4 참조)
5. **재시작 명령 실행**
6. **시작 직후 5초 내 로그 확인**으로 정상 가동 검증
   ```bash
   sleep 5 && tail -n 30 <log_file>
   ```

### 2.2 외부 발언 검증 (Audit) — "봇 발언 신뢰성 0" 원칙

봇·에이전트·디스패치가 "완료했다", "성공했다", "수정했다"고 주장할 때:

- **즉시 신뢰 금지**. 다음 중 **하나 이상**으로 직접 검증한다:
  - **파일 시스템**: `ls -la`, `cat`, `grep`로 실제 변경 확인
  - **프로세스**: `ps`, `lsof`, `netstat`로 실제 가동 확인
  - **로그**: `tail -n 50 <log>`로 최근 출력 직접 확인
  - **외부 동작**: `curl`로 엔드포인트 실제 응답 확인
- 검증 결과를 사용자에게 보고. 봇 주장과 다르면 **명시적으로 차이 지적**.
- 신뢰성 0 원칙은 **자기 자신(클로드)에게도 동일 적용**. "수정했다"고 답하기 전에 직접 검증.

### 2.3 다중 인스턴스 관리

여러 봇 변형(v2, sl, alpha, swing, bot1, bot2 등)이 동시 가동될 때 일괄 출력 표준:

| 필드 | 예시 | 비고 |
|------|------|------|
| 봇 식별자 | `funding_swing_bot_v2` | 풀네임 권장 |
| PID | 40733 | |
| 시작 시간 | 2026-05-22 04:24 | 가동 기간 계산용 |
| 가동 모드 | paper / live / testnet | 필수 |
| 마지막 활동 | 2분 전 | 침묵 감지용 |

동일 포트·리소스 충돌 여부를 시작 전 사전 확인 (`lsof -iTCP -sTCP:LISTEN`).

### 2.4 변경 전 백업 강제

소스·config 교체 전 반드시 타임스탬프 백업:
```bash
cp <target> <target>.before_<change>_$(date +%Y%m%d_%H%M%S)
```
