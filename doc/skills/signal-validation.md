# [스킬] 신호 검증 방법론 (Signal Validation)

## 1. 발동 조건

다음 중 하나에 해당하면 이 스킬을 우선 적용한다:

- 백테스트·상관·lead-lag·펀딩비 등 **정량적 "발견"**을 신뢰·문서화·실거래에 반영하기 직전
- OOS 상관계수·IC(정보계수)가 **비정상적으로 높을 때** (too-good-to-be-true)
- 여러 기간 창(60일/180일/365일 등)으로 **robustness를 주장**할 때
- 봇·에이전트·자기 자신이 **"신호가 검증됐다"고 보고**할 때 (`skills/bot-ops.md §2.2` "신뢰성 0" 연계)
- 발견한 신호를 **전략·필터·stop 조건으로 코드화**하기 직전

이 스킬은 `bot-ops §2.2`의 "봇 발언 신뢰성 0"을 **정량 연구 영역**으로 확장한 것이다. 구체 수치
임계(paper 7일·시드 10% 등)는 `domains/trading/rules.md`가 우선하며, 본 스킬은 **통계적 자기기만을
막는 일반 방법론**만 정의한다.

## 2. 준수 설계 규칙 (레시피)

### 2.1 IC(상관) 타당성 sanity check

발견한 신호의 크기가 현실적인지 **먼저** 의심한다:

- 양질의 단일 신호 IC(예측-수익 상관)는 보통 **0.02 ~ 0.05**. **OOS 상관 |r| > 0.1**이 1년 내내
  유지되면 *알파가 아니라 방법론 아티팩트일 가능성*을 1순위로 점검 (§2.2 ~ §2.4).
- **Sharpe 환산 sanity check**: 그 상관이 진짜면 함의되는 Sharpe가 비현실적(예: > 5)이지 않은가?
  비현실적이면 룩어헤드·겹침 표본을 의심.
- **`r`과 `r²` 혼동 금지**: "금융에선 r²>0.05면 알파" 같은 서술은 틀렸다. 통상 임계값은 **상관(IC)**
  기준이며 `r²=0.05`는 `|r|=0.22`로 이미 거대한 값이다.
- **방향성 정확도 ≈ `0.5 + arcsin(r)/π`** (정규성 가정). 정확도 60%는 `r²`가 아니라 `r ≈ 0.335`에서
  나온다 — "제곱한 0.11 때문에 60%"는 인과 오류.

### 2.2 겹침 표본·자기상관 교정

```
forward-return을 촘촘히 샘플링하면 관측치가 겹친다 → 유효 표본수 ≪ 명목 N
```

- 펀딩비처럼 **고지속성(8h 정산 등) 변수**는 연속 관측치가 독립이 아니다.
- **Bonferroni·일반 p값은 다중검정만 보정**할 뿐 **자기상관 인플레이션은 못 막는다.**
- 교정 수단 **셋 중 하나 이상 필수**: ① 비겹침(non-overlapping) 샘플 ② Newey-West(HAC) 표준오차
  ③ 블록 부트스트랩(block bootstrap). 미적용 시 상관 크기·유의성 둘 다 과대평가로 간주.

### 2.3 중첩창 ≠ 독립 검증

- `60일 ⊂ 180일 ⊂ 365일`처럼 **나중 창이 앞 창을 포함**하면 같은 데이터를 다시 본 것이다.
  "세 기간 모두 일관" 주장은 **독립 3회가 아니라 사실상 1회**.
- 진짜 robustness = **겹치지 않는 구간 분할** + (가능하면) **타 거래소·타 자산** 교차확인.

### 2.4 룩어헤드·정렬·연구자 자유도

- **정렬(alignment)**: `T` 시점에 **이미 확정된** 값으로 `T → T+h` 수익률을 예측해야 한다. 형성 중인
  (예측) 펀딩이나 수익률 구간과 겹치는 정산값을 쓰면 look-ahead.
- **연구자 자유도(researcher degrees of freedom)**: lag(예: 12h)·임계값("비정상으로 높은")·자산을
  여러 개 시도해 고른 것이라면 그 선택 횟수까지 다중검정 보정에 포함했는지 확인.

### 2.5 인과 vs 상관 (벤치마크 대조)

- 상관에 인과 서사("A → B로 정보가 흘러내림")를 얹기 전에 **대체 설명을 배제**한다. 예: "ETH 펀딩이
  BTC를 선행"이 아니라 "시장 전체 레버리지 과열 → 공통 평균회귀"일 수 있음.
- **벤치마크 대조 필수**: 두 자산이 고상관(예: 95%)이면 "A 신호가 B를 예측"은 "A가 A를 예측"과
  거의 같다. 신호가 **B 자체 신호보다 우월**한지(증분 설명력) 보여야 lead-lag 주장이 성립.

### 2.6 자체보고 검증 결합

신호 "검증 통과" 판단은 봇·시스템·자기 자신의 **자체 보고만으로 결정 금지**:

- "OOS 통과" 보고 → 거래·신호 로그 직접 파싱으로 재계산 (`skills/audit-log.md §2.3`)
- 게이트 진입은 `skills/progressive-gate.md §2.5`(자체보고 검증 결합) 결합
- SSOT: `skills/bot-ops.md §2.2` (봇 발언 신뢰성 0 — 자기 자신에게도 동일 적용)

신호가 모든 검증을 통과해도 **실거래 진입은 `domains/trading/rules.md §1.3` 게이트**(backtest →
walkforward → paper 7일 → 시드 10%)를 별도로 따른다.

## 3. 예외 케이스 누적 (지속 업데이트)

- **[2026.05.24]** 초안 도입. "ETH 펀딩비 비정상 고점 → BTC 12h 후 하락(OOS 상관 −0.335)" 발견을
  검토하던 중, (a) −0.335가 정상 IC(0.02~0.05) 대비 과대 → 아티팩트 의심, (b) 12h forward-return의
  겹침 표본·펀딩 고지속성으로 유효 N 과대평가 가능, (c) 60⊂180⊂365 중첩창을 "3회 일관"으로 제시한
  논리 오류, (d) r²↔r 혼동("r²=0.11→60%"는 실제 r=0.335 기여), (e) BTC-ETH 95% 상관 하에서 BTC 자체
  펀딩 대비 우위 미입증 → 인과 미확정. 본 스킬 §2.1~§2.5 각 항목으로 명문화. 상세 발견 기록·후속
  검증 경과는 `domains/trading/rules.md §8` [2026.05.24] 항목과 교차참조.
