# [메타] 스킬·룰 정의 표준 (Skill Define)

이 문서는 `doc/` 하위에 새 스킬·룰을 추가하거나 기존 항목을 수정할 때 적용된다. **다른 모든 파일이 이 표준을 통과해야** 디렉토리에 진입한다.

---

## 1. 발동 조건

다음 중 하나에 해당하면 이 문서를 우선 흡수한다:

- 새 스킬(`skills/*.md`) 또는 도메인 룰(`domains/<name>/rules.md`)을 작성할 때
- 기존 스킬·룰의 핵심 섹션을 변경(추가·삭제·재배치)할 때
- 파일을 **삭제·이름변경·병합**할 때 (영향받는 인용·README 트리·우선순위 일괄 점검 필요)
- 어떤 내용이 `skills/`와 `domains/` 중 어디 속할지 분류가 모호할 때
- 신규 도메인 디렉토리(`domains/<new>/`)를 만들지 결정할 때

### 1.1 경량 변경 vs 중량 변경

모든 변경에 16점 전수 적용은 비효율. 변경 성격에 따라 절차를 나눈다:

| 구분 | 예 | 적용 절차 |
|------|-----|-----------|
| **경량** | 오타·문법 수정 / 단일 예외 케이스 누적 추가 / 기존 인용 보강(링크·섹션 번호 정정) | §7 자기검증만 적용. §2 5점 체크·§3 Spec 7항목 **면제** |
| **중량** | 신규 파일·섹션 신설 / 핵심 섹션 재배치 / 도메인 신설·삭제 / 파일 병합·이름변경 / 우선순위 체인 변경 | 5점 체크 → Spec 7항목 사용자 승인 → 본문 → §7 자기검증 (전 절차 강제) |

판단 모호 시 **중량으로 처리** (안전 측).

---

## 2. 작성 전 5점 체크 (전부 통과해야 진행)

| # | 항목 | 검사 내용 |
|---|------|----------|
| 1 | 중복 | 기존 파일에 동일·유사 내용이 있는가? 있다면 흡수 vs 신설 판단 |
| 2 | 범용성 | 봇 1개·사이트 1개에만 적용되는 케이스인가? 그렇다면 본문이 아닌 해당 파일의 "예외 케이스 누적"에 |
| 3 | 도메인 누수 | trading 룰이 automation/ 안으로 들어가지 않는가? 도메인 경계 점검 |
| 4 | 보안 | 시크릿·자격증명을 평문 노출하는 예시가 있는가? `security/rules.md` §1.1 위반 즉시 차단 |
| 5 | 우선순위 충돌 | 기존 도메인 룰과 충돌하는가? 충돌 시 도메인 룰이 우선 (§6 참조) |

체크 1개라도 실패 → 진행 금지. 사용자에게 보고 후 재설계.

---

## 3. Spec 7항목 (파일 생성 전 사용자 승인 필수)

새 파일을 만들기 전 다음 7항목을 사용자에게 제시·승인받는다:

| # | 항목 | 정의 |
|---|------|------|
| 1 | 이름 | kebab-case. `skill_` 접두사 금지 |
| 2 | 위치 | `skills/<name>.md` · `domains/<name>/rules.md` · 루트 중 하나 |
| 3 | 발동 조건 | 클로드가 이 파일을 펴보게 되는 트리거 (3개 이상 권장) |
| 4 | 핵심 섹션 | 본문 목차 (3~6개) |
| 5 | 다른 파일과의 관계 | 인용·피인용·대체 관계 명시 |
| 6 | 우선순위 | 충돌 시 위/아래 어디인가 |
| 7 | 예외 케이스 누적 정책 | 실전 학습 추가 시 어디에 어떻게 |

**Spec 승인 없이 파일 생성 금지** (중량 변경 한정).

---

## 4. 분류 룰: skills/ vs domains/

판단 기준은 **이름의 품사**가 가장 빠르다:

| 형태 | 위치 | 예시 |
|------|------|------|
| **동사형** (행위) | `skills/<verb>.md` | bot-ops(운영하다), infra-debug(디버깅하다), system-health(점검하다), progressive-gate(게이트 통과), audit-log(로그 남기다) |
| **명사형** (영역) | `domains/<noun>/rules.md` | trading, automation, security |

경계 케이스 판단:

- 한 도메인 안에서만 의미 있는 행위 → 해당 도메인 `rules.md`에 섹션으로 흡수
- 여러 도메인에 걸치는 행위(예: 백업·재시작·검증·게이트·로그) → `skills/`
- 한 도메인의 데이터·상태·표준 정의 → `domains/<name>/rules.md`

### 4.1 양식 표준

| 위치 | 양식 |
|------|------|
| `skills/<name>.md` | **고정 양식**: `§1 발동 조건 / §2 준수 설계 규칙(레시피) / §3 예외 케이스 누적` 3섹션 |
| `domains/<name>/rules.md` | **자유 양식**: 주제별 절을 자유 배치. 단 마지막은 `예외 케이스 누적` 절 고정. **발동 조건 절 불필요** (도메인 진입 시 자동 흡수) |
| `doc/` 루트 (메타) | 본 `skill-define.md`와 같은 자유 양식. 본 문서의 §1~§9 형태가 참조 패턴 |

## 5. 네이밍 규약

- **kebab-case 강제**: `bot-ops`, `system-health` (✅) / `bot_ops`, `botOps` (❌)
- **`skill_` 접두사 금지**: 디렉토리 이름(`skills/`)이 이미 그 역할. 중복.
- **도메인명**: 단수형 명사 권장 (`trading`, `automation`). 복수형(`tradings`)·동명사형(`trading-ing`) 금지.
- **확장자**: `.md` 고정.

---

## 6. 우선순위

충돌 시 다음 순으로 적용:

```
domains/security/rules.md       ← 최상위 (보안은 어떤 룰도 우회 불가)
domains/<other>/rules.md        ← 도메인 룰
skills/*.md                     ← 크로스커팅 스킬
skill-define.md                 ← 메타 (작성 방법론)
```

예시:

- `bot-ops.md`의 lifecycle과 `trading/rules.md`의 paper-first가 충돌 → trading 우선
- `skill-define.md`의 양식 권고와 `security/rules.md`의 시크릿 평문 금지가 충돌 → security 우선

## 7. 자기검증 원칙

본 문서를 따라 새 파일을 만들거나 기존 파일을 변경한 직후 **반드시 직접 검증**한다. "봇 발언 신뢰성 0" 원칙의 단일 진리원(SSOT)은 `skills/bot-ops.md` §2.2이며, 본 절을 포함한 모든 인용은 bot-ops를 가리킨다:

1. **파일이 실제 경로에 생성·갱신되었는가?** (`ls -la <path>` / `get_file_contents <path>`)
2. **README의 디렉토리 트리에 추가·반영되었는가?** (`grep <name> doc/README.md`)
3. **계획 트리 ↔ 실제 트리 diff** — 푸시 직후 `get_file_contents doc/` 1회 호출해 의도한 파일 목록과 실제 목록이 정확히 일치하는지 검증 (§9 [2026.05.23] 사례 — "계획 4+4, 실제 3+3"의 재발 방지)
4. **기존 파일에서 이 파일·섹션을 인용해야 하는 위치가 있는가?** 관련 도메인·skills 5개 정도 grep해 누락 인용 확인
5. 중량 변경의 경우 **5점 체크·Spec 7항목 결과**를 본 파일 또는 PR 설명에 첨부

봇·디스패치·자기 자신의 "완료했다" 발언은 신뢰 금지. CLI / `get_file_contents`로 직접 확인.

## 8. 변경 시 절차

본 파일(`skill-define.md`) 자체의 수정은:

1. 변경안을 사용자에게 제시
2. 영향 받는 기존 파일(`skills/`·`domains/`) 목록 첨부
3. 승인 후 본 파일 변경 + 영향 파일 일괄 갱신
4. 본 §9에 변경 일자·사유 기록
